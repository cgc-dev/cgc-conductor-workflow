0
Scope lock — intake gate
Gate

Classify and lock scope before any work. State: "This request is a bugfix / small change / EPIC-level feature".

  • Bugfix or small change → divert to the Bugfix track
  • EPIC-level feature → continue with full planning
  • Unclear → ask one clarifying question first

Never silently expand a bugfix into a refactor.

2
Implementation cycle
Repeats per EPIC
Loops 2A → 2F for each EPIC. After 2F, return to 2A for the next EPIC, or proceed to Phase 3 when all EPICs are done.
2A
Implement EPIC
implement-subagentLoops

Step 0: Write session state FIRST to docs/session/conductor-state.md with current phase, last completed step, and retry counts.

Spawn implement-subagent with the EPIC number and objective, files/functions, test requirements, autonomous TDD instruction, and the requirement to create the EPIC file at the start.

Validate output: confirm Status, Summary, Files Created/Modified, Tests Written, Test Results, Security-Sensitive Changes, Out-of-Scope Discoveries, Blockers — all present.

  • Missing field → BLOCKED, ask resubmit
  • Security-Sensitive ≠ None → cross-check trigger table, flag for 2D
  • Out-of-Scope ≠ None → log in epics-status.md
  • Blockers ≠ None → STOP and escalate
↓
2B
Review implementation
code-review-subagentRetry → 2A

Spawn code-review-subagent with EPIC objective, modified files, and instruction to verify tests pass and code follows best practices. Verify EPIC docs follow the style guide.

  • APPROVED → proceed to 2C
  • NEEDS_REVISION → increment retry, return to 2A. After 2 retries: STOP and escalate verbatim.
  • FAILED → stop and consult user
↓
2C
Test validation
test-subagentRetry → 2A

Spawn test-subagent with EPIC scope, acceptance criteria, and instruction to run existing tests and write any missing coverage.

  • PASSED → proceed to 2D
  • FAILED / PARTIAL → increment retry, return to 2A. After 2 retries: STOP and escalate verbatim.
↓
2D
Security review
security-subagentRetry → 2A

Spawn security-subagent against OWASP Top 10. Auto-runs whenever changes touch authorization, tokens, user-input persistence, file upload/download, CORS/CSP, or env-var/secrets handling.

  • APPROVED → proceed to 2E
  • NEEDS_REVISION → increment retry, return to 2A. After 2 retries: STOP.
  • FAILED → consult user on critical findings
↓
2E
Return to user for commit
Mandatory stop

Pause and present: EPIC objective, what was accomplished, files changed, review status (code/tests/security all approved or addressed), epics-status.md location.

Write docs/epics/<task-name>-epic-<N>-complete.md. Update docs/epics-status.md. Provide commit message in a plain text code block.

Evidence checklist before presenting: cite Status lines from 2B/2C/2D, confirm EPIC Task Status table all Completed, epics-status.md row updated, session state shows 2E.

Mini-cycle, alternative path
B
Mini-cycle for bugs & small direct changes

For bugs or small direct changes outside a planned EPIC:

  • Log the bug if trigger keywords present ("bug, broken, defect, issue, not working, crash, error, regression, fail")
  • Implement the fix (directly or via implement-subagent)
  • Code review — code-review-subagent for non-trivial fixes
  • Security check if change touches security triggers
  • Tests — test-subagent, confirm passing via verification-before-completion
  • Documentation — documentation-subagent for affected docs
  • Provide commit message only after all blocked-until gates pass
